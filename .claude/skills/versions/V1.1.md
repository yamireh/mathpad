# MathPad — V1.1 Scope

Post-launch update bucket: enhancements that ship in a normal app release after
V1 is live. Tracks what's committed for V1.1 vs. merely a candidate.

## Committed for V1.1

### Remote config loader (best-effort, non-blocking)

A tiny JSON of tunable values the app reads on launch, so we can adjust a few
parameters without resubmitting to the stores.

**Behavior (this is what keeps it within SPEC — no backend, no data collection):**
- On launch, **best-effort** fetch of one small static JSON from a public URL.
- **Never blocks app start.** The fetch is async; the UI never waits on it.
- **Bundled defaults ship in the app** — first launch, offline, or a failed/slow
  fetch all just use the in-app defaults. Nothing breaks without the network.
- Cache the last good fetch locally; fall back cache → bundled defaults.
- It's a static file read, no accounts, no analytics, no payload sent — so the
  offline/no-backend posture holds. If the call fails, that's fine by design.

**Config candidates (values only — never executable logic, per store policy):**
- Timing tunables — e.g. hint step delay, bring-down drop duration, hand-cursor
  pacing (the kind of `ms` knobs we've been hand-tuning in `AnswerBox` /
  `DivisionDraftGrid` / `useSolver`).
- **`minSupportedVersion`** → a force-update gate: if the installed build is
  older, show a blocking "Please update" screen with a store link. (Allowed
  pattern; Apple/Google don't auto-update, we just gate the UI.)
- Feature flags / kill-switches for risky features.

**Design rules at build time:**
- Versioned, validated schema — clamp ranges and ignore unknown keys so a bad
  edit can't brick the app.
- Single source of truth: a `lib/remoteConfig` that merges bundled defaults ←
  cached ← fetched, exposing typed getters the rest of the app reads.
- Host as one static file (microclouds.ca / a CDN / GitHub raw). Mind CDN
  propagation delay.
- Re-check the privacy label / Data-Safety wording at implementation time to
  describe the single best-effort static fetch accurately (no data collected).

## Rules for working in V1.1 scope

1. Remote config is **best-effort and additive** — it must never become a hard
   dependency. If the app can't run fully on bundled defaults with no network,
   that's a bug.
2. Config carries **values/flags only**, never downloaded logic (store policy).
3. Don't expand V1.1 beyond what's in "Committed" without the user moving an item
   up from Candidates.
